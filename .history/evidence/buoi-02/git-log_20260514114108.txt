Key Decisions Recorded
======================

1. Queue Async Transport (not REST sync)
   - Chosen because: Core Business fire-and-forget, Analytics batch processing
   - Timestamp: RFC3339 (enable cross-TZ consistency)
   - Partition: by alertId (maintain FIFO per alert)

2. Idempotency Strategy
   - Provider: guarantee unique eventId (UUID v4)
   - Consumer: maintain 30-day cache, dedup before processing
   - Rationale: At-Least-Once delivery requires consumer-side handling

3. Out-of-Order Resilience
   - Alert.resolved may arrive before alert.created (network variance)
   - Consumer buffers unmatched resolve events for 5 minutes
   - Reconciliation happens when alert.created arrives
   - Orphaned events (no match after 5min) logged for investigation

4. Schema Versioning
   - eventSchemaVersion field included in all events (v1.0)
   - Backward-compatible changes: add optional fields only
   - Breaking changes: increment to v2.0, support both versions during migration
