## Decisions Made (Rationale in negotiation-log.md)

✓ Queue Async (not REST sync) — chosen for fire-and-forget, batch processing  
✓ RFC3339 timestamps — cross-TZ consistency  
✓ API Key authentication — simplicity vs JWT for internal service  
✓ At-Least-Once delivery — chose to push dedup to Consumer  
✓ FIFO per alertId — compromise between global order (costly) and no order (risky)  
✓ 5-min out-of-order buffer — empirically sized for typical network variance  
✓ 24h queue retention — aligns with operational SLA, SSD cost, disaster recovery needs  

## Lessons Learned

1. **Event schema design first:** Payload format agreement is hardest; code follows (done ✓)
2. **Dedup is Consumer's job:** At-Least-Once requires consumer-side idempotency (agreed ✓)
3. **Versioning from day 1:** eventSchemaVersion field prevents future breaking changes (added ✓)
4. **Explicit constraints matter:** Rate limits, timeouts, retention must be written down (negotiation-log ✓)
5. **Out-of-order is harder than in-order:** Alert causality (create before resolve) needs buffer logic (designed ✓)
